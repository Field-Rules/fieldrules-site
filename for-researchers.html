The Approach
Human reasoning as a structured data type

FieldRules captures domain expert judgment as structured IF/THEN/BECAUSE rules — a reasoning substrate that sits between your product and the AI model. The BECAUSE field is the critical innovation: it carries the human rationale that no model can generate from training data alone.

01 · INTERCEPT
Capture at the point of action

When a domain expert resolves an edge case, FieldRules intercepts the conditional logic before it gets lost. A 45-second Slack card surfaces the decision in context — not a documentation sprint weeks later, when the expert confabulates a post-hoc rationalization instead of reporting the actual reasoning.

02 · STRUCTURE
IF/THEN/BECAUSE format

The rule captures three things: the condition (IF), the action (THEN), and — critically — the reasoning (BECAUSE). The BECAUSE field is what transforms a checklist into transferable judgment. It's what enables the AI to handle novel situations correctly, not just template cases. Nisbett & Wilson (1977) showed verbal reports on mental processes are often inaccurate. Contrastive prompting — "why A over B?" — reveals actual decision factors.

03 · DEPLOY
Test-time enrichment

Rules enter the product as structured context at inference time. Not fine-tuning. Not prompt engineering. A governed reasoning library that the model operates within — auditable, attributable, versionable. Ovadia et al. (2024) showed retrieval-based methods outperform fine-tuning for domain-specific knowledge when reasoning transparency is required.

Key Insight
The BECAUSE field is the entropy source

AI-generated rules look plausible individually, but the distribution collapses — the same generic reasoning, the same safe defaults. Andrej Karpathy calls this "silent entropy collapse." The human-authored BECAUSE field is what keeps the library from collapsing into the industry average. It carries the proprietary operational logic that makes a company's AI behave like that company, not the generic case.

Methodology
Controlled 4-condition design with blind scoring

Each domain generates 3 base scenarios plus harder transfer scenarios (ambiguity + domain transfer). Every scenario is run under 4 conditions: (A) Rule + BECAUSE, (B) Rule only, (C) Playbook/documentation, (D) Transcript/raw context. The model, prompt structure, and scenario text are identical across conditions — only the context content varies. Scoring is blind: the evaluator sees conditions labeled A/B/C/D with no description of what each contains. Five dimensions are scored: correctness, novel twist handling, reasoning depth, specificity, and confidence calibration. Cost: ~$3–8 per full run.

Architecture
Training-time investment, test-time returns

The structural argument for why test-time structured context beats training-time fine-tuning for domain-specific reasoning — and why the economics favor FieldRules' approach.

Training-Time Fine-Tuning
Bake knowledge into weights

Knowledge becomes opaque once embedded. Can't audit which rule was applied. Can't update without retraining. Can't attribute decisions to human experts.

  • ✕ Opaque — can't trace decisions to sources
  • ✕ Expensive to update — full retraining cycle
  • ✕ No attribution — expertise disappears into weights
  • ✕ Drift-prone — model behavior changes with each retrain
  • ✕ Domain leakage — training data from one client can bleed
VS
Test-Time Structured Context
Inject reasoning at inference

Rules are structured context the model operates within at query time. Auditable, versionable, attributable. The model improves — the library becomes more valuable, not obsolete.

  • ✓ Auditable — every decision traces to a specific rule and author
  • ✓ Instant updates — practitioner changes a rule, it's live
  • ✓ Attributed — every rule carries the expert's name
  • ✓ Model-agnostic — switch providers, keep your library
  • ✓ Compounds — better models extract more value from same rules
The Tesla Analogy
Training-time sensors, test-time deployment

Tesla's self-driving uses expensive lidar sensors at training time but deploys vision-only cameras at test time — an arbitrage on sensor cost. FieldRules works the same way: the expensive resource is your domain expert's judgment at authoring time — 45 seconds confirming a BECAUSE field. At query time, that judgment serves every AI decision, every new hire, every edge case — without the expert in the loop. The human investment happens once per rule. The AI return happens at every query, forever.
