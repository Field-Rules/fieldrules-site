The Thesis

The model is interchangeable.
The harness is durable.

LangChain jumped from Top 30 to Top 5 on Terminal Bench — same model, different harness. OpenAI shipped Codex with the explicit principle that the harness is the product.

FieldRules is the harness layer for business logic. Other tools provide structural correctness — whether code compiles, passes tests, matches a format. FieldRules provides reasoning correctness — whether the AI applied your company's actual judgment, not the statistical average.

Fine-tuning changes how the model thinks. Guardrails block what the model says. FieldRules tells the model what to do — and why — based on human-confirmed operational rules. That's a different layer.

Integration

How your code reads from the rule library.

Semantic retrieval at inference time. Your code sends a query with context — FieldRules returns the matching rule, the BECAUSE, the author, the version. Treat it like a config read, not a database query.

// Before: hardcoded business logic const shouldEscalate = ticket.waitTime > 30; // After: FieldRules semantic retrieval const rule = await fieldrules.query({ context: 'p0_ticket_escalation', ticket: ticket.metadata }); // → { rule_id: 'FR-2103', action: 'escalate_senior', // threshold: 30, because: '...', author: 'Jordan Chen', // version: 3, confirmed: '2026-03-14' }

The @fieldrules query is a semantic match, not a key lookup. It understands context. When your codebase has 200 business rules externalized this way, you've eliminated 200 deploys per rule change — and you have an audit trail for every decision your system makes.

Every AI governance tool tells you what rules exist. FieldRules is the only one that tells you whether your rules are actually thinking.
