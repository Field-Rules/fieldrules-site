when business rules change.

A domain expert changes how renewals work. It gets communicated in a team sync. Maybe someone updates a Confluence page. But the code still enforces the old rule — and the AI agent trained last quarter has a different version entirely. Three sources of truth. Zero provenance.

Architecture

The harness layer for business logic.

FieldRules captures operational rules from domain experts, structures them as IF/THEN/BECAUSE blocks with provenance metadata, and makes them queryable by your code, your AI agents, and your tooling. The model is interchangeable. The harness is durable.

01 · Intake
Rule surfaces
Domain experts capture rules from Jira tickets, Slack threads, and operational workflows — anywhere judgment is already being applied.
02 · Structure
IF / THEN / BECAUSE
Each rule is decomposed into structured context: conditions (IF), actions (THEN), and human reasoning (BECAUSE) — the part no model can invent.
03 · Confirm
Expert attribution
A named practitioner confirms each rule. Author, role, date, and version are immutably attached. Every rule has a human on the other end.
04 · Govern
Rule library
Confirmed rules populate a structured library — versioned, searchable, and available through a typed API. Rule URIs are permanent identifiers.
05 · Consume
Five decision surfaces
Rules govern decision routing, approval gates, workflow orchestration, agent constraints, and test specs. One library, five integration points.

Rule URI Scheme

Reference rules like you reference code.

Every confirmed rule gets a permanent URI: fieldrules://rule/{uuid}. Use it in code comments, PR descriptions, incident reports, and spec documents. The URI never changes post-MVP — it's a founding constraint.
